# Web Antenna 仕組み解説

このドキュメントでは、Web Antenna がどのような仕組みで動いているかを解説します。

---

## 全体像

Web Antenna は HTML ファイル1つで完結しています。サーバーもデータベースも不要で、ブラウザだけで動きます。

```
ブラウザ（antenna.html）
  ├─ 画面表示（HTML / CSS）
  ├─ 操作・ロジック（JavaScript）
  └─ データ保存（localStorage）
```

データはすべてブラウザの localStorage に保存されます。ページを閉じても登録サイト・スナップショット・更新状態が消えない理由はこれです。

---

## なぜプロキシが必要なのか

ブラウザには「CORS（Cross-Origin Resource Sharing）」というセキュリティの仕組みがあります。

> **CORS とは：** ブラウザが「別のドメインのデータを勝手に取得してはいけない」というルールを設けている仕組み。`file://` で開いた HTML から外部サイトを直接取得しようとすると、ブラウザがブロックします。

監視対象は外部サイトのHTMLページであるため、そのまま取得しようとするとCORSエラーになります。これを回避するために「CORSプロキシ」を使います。

```
直接取得（ブロックされる）:
  ブラウザ → example.com/  ✗ CORSエラー

プロキシ経由（成功）:
  ブラウザ → プロキシサーバー → example.com/
             （同一オリジン扱い）
```

---

## 使用しているプロキシ一覧

Web Antenna は4つのプロキシを使います（RSS Reader と異なり、direct・rss2json は使用しません）。

| 名前 | 説明 | 特徴 |
|------|------|------|
| `allorigins-raw` | allorigins.win 経由（生テキスト返却） | 汎用的、安定性が高め |
| `allorigins-json` | allorigins.win 経由（JSON形式） | allorigins の別モード。nullチェックあり |
| `codetabs` | codetabs.com 経由 | 一部サイトで有効 |
| `corsproxy` | corsproxy.io 経由 | 上記が失敗した場合のフォールバック |

---

## ページ取得の流れ（fetchPage）

サイトを1件確認するときのHTMLページ取得処理フローです。

```
fetchPage(siteUrl) の処理

1. キャッシュ確認
   └─ このサイトで前回成功したプロキシが記録されていれば、それを最初に試す
      ├─ 成功 → そのまま返す（高速）
      └─ 失敗 → キャッシュを消してフォールバックへ

2. 全プロキシを並列で試す（Promise.any）
   ├─ allorigins-raw を試す  ─┐
   ├─ allorigins-json を試す  │ 同時に全部リクエスト
   ├─ codetabs を試す        ─┘ 最初に成功したものを採用
   └─ corsproxy を試す

3. 成功したプロキシ名をキャッシュに保存
   └─ 次回はそのプロキシから先に試す（高速化）

4. すべて失敗 → エラー
```

**429（レート制限）の対処：** プロキシが429を返した場合、そのプロキシ名をセッション中ブラックリスト（`rateLimitedProxies`）に追加し、以降の取得で除外します。メモリ上の `Set` に保持されるため、ページを再読み込みするとリセットされます。

---

## テキスト抽出の仕組み（extractPageContent）

取得したHTMLからノイズを除去し、本文テキストのみを抽出します。

```
extractPageContent(html) の処理

1. DOMParser でHTMLを解析（仮想DOMを構築）

2. ノイズ要素を削除
   └─ script / style / noscript / iframe / svg
      nav / footer / header / aside / form
      role="navigation" / "banner" / "contentinfo" / "complementary"

3. ページタイトルを取得（<title> または最初の <h1>）

4. body.textContent を取得
   └─ 連続する空白を1つに圧縮
   └─ 先頭50,000文字に切り詰め（localStorage 容量節約）
```

広告・メニュー・フッターなどのノイズを除いた「本文らしいテキスト」だけを比較対象にすることで、誤検知を減らしています。

---

## 変更検出の仕組み（SHA-256ハッシュ）

テキスト全体を毎回保存するのではなく、ハッシュ値（要約）を比較することで変更を検出します。

```
computeHash(text) の処理

1. TextEncoder でテキストをバイト列に変換
2. Web Crypto API（crypto.subtle.digest）で SHA-256 ハッシュを計算
3. 16進数文字列（64文字）として返す

比較:
  前回のハッシュ ≠ 今回のハッシュ → テキストが変化した（更新あり）
  前回のハッシュ = 今回のハッシュ → 変化なし
```

**なぜハッシュを使うのか：**

テキストを丸ごと比較することも可能ですが、ハッシュを使うと「変化した/していない」の判定だけなら64文字の比較で済みます。ただし、具体的にどこが変わったかはテキスト本体を比較する必要があるため、スニペット生成用にテキスト全体も保存しています。

---

## 変更箇所の抽出（findSnippet）

更新が検出されたとき、「どこが変わったか」を示すスニペットを生成します。

```
findSnippet(currentText, prevText) の処理

1. 先頭から1文字ずつ比較し、最初に異なる位置（pos）を特定
2. pos から 60文字手前を切り出し開始位置とする（単語境界に近い位置で切り出し）
3. そこから最大 280文字を抜粋
4. 抜粋が空文字になった場合（差分箇所の前後にテキストがない等）は先頭200文字を返す
```

---

## 確認の流れ（checkSite）

登録サイト1件を確認するときの完全な処理フローです。

```
checkSite(index) の処理

1. 同一サイトの確認が既に進行中なら、その完了を待って終了（重複実行防止）

2. ステータスを "checking" に更新（赤スピナー表示）

3. fetchPage でHTMLを取得（最大30秒でタイムアウト）

4. extractPageContent でテキスト抽出・タイトル取得

5. computeHash でハッシュ計算

6. ハッシュ比較
   ├─ prevHash が未設定（初回確認）
   │  → ステータス "ok"（スナップショット保存のみ、「更新あり」にはならない）
   │
   ├─ prevHash ≠ 今回のハッシュ（変化あり）
   │  → findSnippet でスニペット生成
   │  → lastChangedAt を更新
   │  → ステータス "changed"（「更新」バッジ表示）
   │
   └─ prevHash = 今回のハッシュ（変化なし）
      → ステータス "ok"

7. prevHash・prevText・lastCheckedAt を更新して保存
```

---

## 全確認の仕組み（checkAll）

「全確認」ボタンまたは `r` キーで全サイトを確認します。

```
checkAll() の処理

並列数: 3（セマフォプール方式）

  ┌───────────────────────────────────────────────┐
  │ キュー: [サイトA, B, C, D, E, F, G, ...] │
  └───────────────────────────────────────────────┘
           ↓ 3本のworkerが同時スタート
  Worker1: A完了 → すぐ次のDを開始
  Worker2: B完了 → すぐ次のEを開始
  Worker3: C完了 → すぐ次のFを開始
  ...（常に3本が走り続ける）
```

**なぜ3件並列か：**

RSS Reader（10件並列）と異なり、CORSプロキシへのアクセスが集中しすぎないよう少なめに設定しています。外部サイトとプロキシの両方に配慮した値です。

---

## 並び順の仕組み（sortedSites）

サイト一覧の表示順は、更新が検出された日時でソートされます。

```
sortedSites() の並び順ルール

1. lastChangedAt（最後に更新が検出された日時）が新しい順
2. 一度も更新が検出されていないサイトは addedAt（登録順・古い順）で末尾に固定

「既読にする」を押しても lastChangedAt は変わらないため、並び順は変わらない
```

---

## データの保存先（localStorage）

すべてのデータはブラウザの localStorage に保存されています。

| キー | 内容 |
|------|------|
| `antenna-sites` | 登録サイト一覧（URL・ラベル・ハッシュ・テキスト・更新日時など） |
| `antenna-proxy-cache` | サイトごとの成功プロキシ名 |
| `antenna-theme` | 表示テーマ（dark / light） |

`antenna-sites` には各サイトの `prevText`（抽出テキスト全文）も含まれるため、登録サイトが増えると localStorage の使用量が増えます。テキスト上限を 50,000 文字に制限しているのはこのためです。

---

## まとめ：サイトを追加してから更新検出されるまでの全体フロー

```
1. URLを入力して「add」
   ↓
2. fetchPage で複数プロキシを並列試行しHTMLを取得
   ↓
3. extractPageContent でノイズを除去・テキスト抽出
   ↓
4. computeHash でSHA-256ハッシュを計算
   ↓
5. 初回はスナップショット（hash + text）を保存して完了
   ↓
   〜 次回の確認時 〜
   ↓
6. 同じ手順で取得・抽出・ハッシュ計算
   ↓
7. 前回ハッシュと比較 → 差異あり
   ↓
8. findSnippet で変更箇所の抜粋を生成
   ↓
9. lastChangedAt を更新、ステータスを "changed" に
   ↓
10. データを localStorage に保存、「更新」バッジ表示
```
